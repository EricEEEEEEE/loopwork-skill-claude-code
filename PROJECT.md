## 0. 一句话定位

**一个拖进 Claude Code 就能用的引导型 skill，分上下两个半场：上半场，以向导身份带一个不会编程的人走完从想法到能用的软件的全过程；下半场，把他和他的项目一起装进循环——从此他出方向、AI 跑圈，项目永续生长。用户全程只做三件事：回答问题、看结果、点批准。**

产品的本质不是「帮小白做出一个项目」，而是**「Loop 工作法的自动安装器」**。做出第一版只是教学关卡，真正的交付物是一个转起来的循环：**点火 → AI 跑批 → 收货 → 续单 → 再点火**。项目不是做完的，是转起来的（详见 §4.8 下半场）。

30 秒版本：市面上「小白友好的 AI 建站产品」（Lovable / Bolt / Replit Agent）和「Claude Code 里的工程级工作流」（superpowers / spec-kit / BMAD）是两条从未交汇的线——前者小白友好但封闭收费，后者强大但假设用户是工程师；而且两边做的都是「一次性交付」。Loopwork Skill 站在交汇点上，还多走一步：**把一套经过验证的工程纪律（规格驱动 + 测试先行 + 自主循环 + 人工验收）翻译成小白能懂的白话装进一个文件夹，并且以「循环」而不是「交付」为终态。**

## 1. 为什么做这个

### 1.1 问题

Claude Code 有能力独立完成整个项目，但小白用不动它，原因不是模型不够强，而是：

1. **不知道说什么**——面对空白输入框，说不清需求（"帮我做个记账的"之后就没词了），而模糊指令正是产出灾难的头号原因；
2. **不知道流程该长什么样**——不知道「先写清楚要什么、再动手」这个工程常识，直接开写，越改越乱；
3. **没有能力验收**——看不懂代码，无法判断"AI 说做完了"是不是真做完了；
4. **不知道现在在哪、下一步干嘛**——项目一大，多次会话之后彻底迷路；
5. **踩安全的坑**——误删文件、密钥泄露、循环烧钱、不敢点权限弹窗。

这五个问题每一个都有成熟解法，但**没有人把解法组装起来给小白**。

### 1.2 空位（2026-07 生态调研结论）

我们对 GitHub 生态做了四路调研（官方 skill 规范 / spec-driven 框架 / 自主循环纪律 / 小白引导先例），结论：

- 官方 18 个 skill、四个 awesome 清单里，**不存在**「小白拖入即用、引导完成整个项目」的 skill；
- 最接近的工程框架 superpowers / spec-kit / BMAD 全部**面向工程师**，且都要求终端安装 CLI 工具链（Python/Node/uv），这个前置步骤本身就是小白的第一道劝退墙；
- 最接近的小白产品 Lovable / Bolt 是**封闭付费 Web 产品**，不是 Claude Code 生态的东西。

**这是一个空位，不是一个红海。** 同时，每个部件都有经过实战验证的先例可抄，我们不发明任何未经验证的新概念（详见附录 A 设计依据）。

### 1.3 我们的差异化

1. **零安装门槛**：一个文件夹拖进 `~/.claude/skills/`，官方一等公民安装方式，不碰终端、不装依赖；
2. **白话词典作为界面语言**：测试叫「考题」、提交叫「存档」、PR 叫「成果包」、待决事项叫「问题本」、代码审查叫「判卷员」——整套比喻让小白不需要学任何黑话就能理解工程纪律**为什么存在**；
3. **纪律是硬的，不是嘴上的**：验收靠脚本跑出的 exit code 而不是 AI 的自述；红线靠 hook 物理拦截而不是提醒；这是行业用 68 个失败案例换来的教训（"文本规则会被模型读了、懂了、压力下照样违反"）；
4. **方向永远在用户手里**：AI 负责执行的全部脏活，但每个阶段的方向确认、每个不可逆动作的批准，永远是用户的——「执行自主，方向人定」；
5. **以循环为终态，不以交付为终态**：所有向导型产品在「项目做完」时散场；本 skill 在「第一版做完」时才进入正题——把用户带进「出方向 → AI 跑圈 → 收货 → 续单」的永续循环。这是方法论名字 Loopwork 的本意。

## 2. 目标用户

### 2.1 核心画像：「有想法的小白」

- 有一个具体的软件想法（工具、网站、小游戏、自动化脚本……）；
- 会安装应用、会打字聊天，**不会写代码**；
- 不懂 git、不懂终端、不懂部署，听到 "agentic" 这类词会直接关页面；
- 已经装好了 Claude Code（或者有人帮他装好了）；
- 愿意花几个小时到几天，跟着引导把想法变成真的。

### 2.2 次级画像

- **会一点但不体系的人**：写过一些代码，但没有工程流程概念，项目总是烂尾；skill 的流程纪律对他们同样有效，且可以选择性关闭部分白话解释；
- **想给家人朋友装一个的工程师**：自己用专业工具，但想让身边的小白也能安全地用 Claude Code 做东西。

### 2.3 明确的非目标用户（v1）

- 需要多人协作、云端会话、PR 工作流的团队 —— 那是专业版（eric-loop-standard）的领域；
- 要改造大型存量代码库的人 —— 循环方法论的适用边界是全新项目和小型项目（Ralph 技术的原作者原话："绝不会在存量代码库上用"），v1 只对小型存量项目提供受限支持；
- 想学编程的人 —— 本 skill 的目标是**做出东西**，不是教学（虽然过程中自然会学到流程）。

## 3. 产品原则

继承自 Loopwork 方法论（上层）＋小白化新增（下层）：

| # | 原则 | 一句话解释 |
|---|---|---|
| P1 | 文件即记忆 | 项目的全部状态活在文件里，不活在对话记忆里；换会话、隔一周回来，都能无损续接 |
| P2 | 先说清楚，再动手 | 规格文档没定稿不写代码；动手前把计划摊开给用户看，批准了才开始 |
| P3 | 考题先红后绿 | 先写测试并亲眼看它失败，再写实现让它通过；实现期间绝不改考题 |
| P4 | 执行自主，方向人定 | 循环内 AI 全自动；阶段方向、花钱、删除、发布，永远等用户拍板 |
| P5 | 验收看证据，不看自述 | 「做完了」必须有 exit code、能点开的页面、逐条对上的清单为证 |
| P6 | 卡住不停机 | 需要用户拍板的事写进问题本，跳过继续干别的，攒着一起问 |
| P7 | 白话优先 | 每个术语首次出现必带比喻解释；能不提黑话就不提 |
| P8 | 一次只问一个问题 | 不确定时逐个问、选择题优先、永远给「不知道，你决定」的出口 |
| P9 | 用户永远知道自己在哪 | 每阶段结束自动报告进度和下一步；随时可问「我在哪」 |
| P10 | 硬纪律不靠自觉 | 红线（改考题、删文件、动密钥）用 hook 物理拦截，文本规则只是第一层 |
| P11 | 清单空了 ≠ 项目完成 | 一批任务跑空只意味着该续单了；项目是转起来的，不是做完的 |

## 4. 用户旅程与功能详述

这是本文档的核心章节。旅程分上下两个半场：**上半场「首航」是一条直线**（七个阶段 + 小任务旁路），带用户从想法走到第一个能用的版本；**下半场「循环模式」把直线卷成一个环**——此后项目的每一次生长都是绕环一圈，永不散场。上半场是教学关，下半场才是本体。

```
【上半场 · 首航（直线，走一次）】

安装（拖文件夹，一次性）
   ↓
Stage 0 开场与体检 ──小任务？──→ 快速通道（跳过 1-3，直接进迷你循环）
   ↓
Stage 1 想法访谈        →  产出：项目简介 PROJECT.md
   ↓
Stage 2 白话规格＋项目规矩  →  产出：spec.md ＋ rules.md
   ↓
Stage 3 摊开计划        →  产出：plan.md ＋ tasks.md（用户批准后才动代码）
   ↓
Stage 4 循环执行 ⟲      →  一轮一条任务：考题红→实现→绿→存档→勾掉
   ↓                       （要拍板的进问题本，跳过不停机）
Stage 5 验收           →  用户亲手点一遍；通过→回 Stage 4 取下一批 / 清单空→Stage 6
   ↓
Stage 6 首航收尾（可选上线） →  逐步护送：备份、说明书、部署
   ↓
═══ 进环仪式：直线到此卷成环 ═══

【下半场 · 循环模式（圆环，转无限次）】

        ┌──── 点火（你，30 秒）────┐
        │                          │
   续单（你出方向，5 分钟）      跑批（AI 自主，挂机）
        │                          │
        └──── 收货（你，3-10 分钟）─┘
```

### 4.0 安装与触发

**安装**：用户（或帮他的人）把 `loopwork/` 文件夹放进 `~/.claude/skills/`。完成。没有第二步。

**触发**，两种方式：
- **自动接住**：用户在 Claude Code 里说出任何形如「我想做个 XX，但我不会编程」「帮我把这个想法做成一个应用」的话，skill 自动激活——小白不需要知道斜杠命令的存在；
- **显式调用**：输入 `/loopwork` 直接开始。

**触发后的第一句话**（示意基调）：

> 「收到，我来当你的向导。接下来的流程我会一步步带你走，你全程只需要做三件事：**回答我的问题、看我给你的结果、对重要的事点头**。写代码、跑测试、存档这些脏活全部归我。先从聊聊你的想法开始——这一步大概 10 分钟。」

### 4.1 Stage 0 · 开场与体检

**目的**：确认环境可用，征得开工同意，判断任务规模。

skill 做的事：
1. **自我介绍＋约定**：一段话讲清楚流程全貌（上面那张图的白话版）和用户的三件事；
2. **征得同意**：明确询问「要在哪个文件夹建项目 / 可以开始吗」——**未经同意不碰用户文件系统**；
3. **静默体检**：检查 git 是否可用、工作目录是否干净、是否已有旧项目状态文件（有→走续接流程，见 §5.2）；体检结果用一句话报告，不展示技术细节；
4. **规模判断**：这是完整项目，还是一个小改动？——「改个按钮颜色」不配走七阶段（生态调研里所有评测对 spec-driven 框架的头号抱怨就是「大锤砸坚果」）。小任务走**快速通道**：直接确认需求 → 迷你计划（3 行内）→ 动手 → 验收，全程一个会话搞定。

**用户做的事**：点头，或者说换个文件夹。

**退出条件**：环境可用＋用户同意＋规模已判断。

### 4.2 Stage 1 · 想法访谈

**目的**：把「一句话想法」变成一份双方都点头的项目简介。

skill 做的事：
1. 以采访者身份提问，**一次只问一个问题**，用选择题＋「其他」自由输入的形式（这是平台原生的提问格式），推荐项排第一并标注；问题围绕六件事：给谁用、解决什么问题、必须有什么、明确不要什么、什么算成功、有什么绝对不能碰的红线；
2. 每个问题都给「不知道，你来定」的选项——选了就由 skill 给出行业常规默认值并说明理由；
3. 问答期间实时把答案写进草稿；**最多一轮 5-8 个问题**，问完立刻汇总，绝不车轮战；
4. 产出 ≤300 字的**项目简介**（存进项目的 `PROJECT.md`），念给用户听，逐段确认；
5. 有歧义处不静默假设——写成显式的「待澄清」标记，标记不清零不进入下一阶段（抄自 spec-kit 的 `[NEEDS CLARIFICATION]` 门禁机制）。

**对话示意**：

> skill：「这个记账工具主要给谁用？① 就你自己（推荐，最简单）② 你和家人 ③ 打算给很多人用 ④ 其他」
> 用户：「就我自己」
> skill：「明白，那就不需要注册登录系统了——少一大块复杂度。下一个问题……」

**用户做的事**：回答问题（选选项或随口说），最后对简介点头。

**退出条件**：项目简介定稿、零「待澄清」标记、用户明确确认。

### 4.3 Stage 2 · 白话规格＋项目规矩

**目的**：把简介展开成「做成什么样才算对」的验收清单，同时立好本项目的红线。

skill 做的事：
1. 生成 `spec.md`（**白话规格**）：每条功能都写成「用户视角的可验收句子」——不是「实现 CRUD 接口」，而是「打开页面能看到本月支出总额；点『记一笔』，填金额和分类，保存后列表里立刻出现这条」。**小白看不懂技术规格，但看得懂这种句子，而验收清单未来就是他亲手点一遍的脚本**；
2. 生成 `rules.md`（**项目规矩**，宪法的小白版），默认内置六条硬规矩：①不删除用户任何既有文件 ②不花任何钱 ③不对外发布 ④密钥和密码永不写进代码 ⑤考题写好后实现期间不许改 ⑥每完成一条任务立刻存档——外加访谈中用户说的项目红线；
3. 规格逐节念给用户确认，每节 200-300 字、节后确认一次（不整篇轰炸）；
4. 控制篇幅：规格有硬性上限，防止「看起来专业实则空洞」的文档膨胀（评测实锤：某框架为 689 行代码生成过 2577 行文档）。

**用户做的事**：读白话验收句，改想法、提意见，最后点头。

**退出条件**：spec.md 定稿＋rules.md 确认。

### 4.4 Stage 3 · 摊开计划

**目的**：动手前，把「接下来会发生什么」一次性摊开（借鉴 Lovable Plan Mode 的产品实践：批准了才开始生成）。

skill 做的事：
1. 生成 `plan.md`：技术选型（附一句白话理由：「选 X 因为它最常见、出问题最好修」）、要建哪些文件、大概结构——**技术细节收在折叠区，主体永远是白话**；
2. 生成 `tasks.md`：任务清单，每条任务 = 一个用户可感知的进步（「做完这条你就能看到 XX」），按顺序排列，每条都标注对应的考题；给出**总步数和每步的时间预估**；
3. 向用户摊牌：「一共 N 步，预计首个能点开的版本在第 M 步出现。批准就开工。」

**用户做的事**：看计划（不用懂技术区），批准或调整优先级（「我想先看到界面」→ skill 重排任务）。

**退出条件**：用户明确批准。**这是代码动工前的最后一道人工闸门。**

### 4.5 Stage 4 · 循环执行（发动机舱）

**目的**：自主干活。这是 Loopwork 方法论的核心，用户在这个阶段最闲。

**每一轮的固定节奏**（一轮 = 一条任务）：

1. 从 `tasks.md` 取第一条未完成任务；
2. **先写考题并亲眼跑红**——确认考题真的能考住人（考题从 spec.md 的验收句直接翻译而来）；
3. 写实现；
4. 跑全部考题：红→自己修到绿；绿→**存档**（git commit，对用户解释为「存档点，随时能回到这一刻」）、在 `tasks.md` 勾掉这条、更新进度文件；
5. 循环中撞到需要用户拍板的事（选方案 / 要花钱 / 碰红线）→ **写进问题本 `BLOCKED.md`，跳过，继续下一条**——绝不停机干等；
6. 每轮结束一行进度播报：「✅ 3/12 · 记一笔功能完成 · 下一条：支出列表」。

**批次与检查点**：连续执行 N 条（默认 5，可调）或撞到「用户可感知的里程碑」（首次能打开页面）就停下来，进 Stage 5 验收。**轮次由外部计数器控制，不靠模型自己数**（官方 ralph-wiggum 插件文档明确承认：模型自数不可靠，硬上限才是真保险）。

**失败处理**：某条任务连续失败 3 次 → 不再傻重试，转入**诊断模式**（读日志、找根因、修环境），诊断后再试一次；仍失败 → 写进问题本，跳过。

**纪律硬件**（详见 §6）：循环全程，考题文件、规格、项目规矩受 hook 保护——AI 物理上改不了，绕道 Bash 改也会被拦。

**用户做的事**：什么都不用做。可以去喝咖啡。

### 4.6 Stage 5 · 验收（用户的主场）

**目的**：让看不懂代码的人可靠地验收。原则：**小白不读 diff，小白点鼠标**。

skill 做的事：
1. 把应用跑起来（本地启动），给用户一个能点开的地址或窗口；
2. 给出**照着点就行的验收脚本**——直接来自 spec.md 的验收句：「① 打开页面，应该看到本月总额 ② 点『记一笔』填 50 元午饭，列表应该立刻出现 ③ 刷新页面，这条记录应该还在」；
3. 自己先用只读判卷员跑一遍预检（考题全绿？有没有碰红线？声称完成的和实际改动对得上吗？N 对 M 逐条点名），把结论浓缩成一句话给用户；
4. 收集反馈时引导具体化：用户说「不好看」→ 追问「哪个部分？① 颜色 ② 字太小 ③ 排版乱 ④ 其他」——模糊反馈是修复无效循环的根源；
5. 里程碑庆祝：首次跑通、过半、全部完成，都明确庆祝（「🎉 你的应用第一次跑起来了——很多人到不了这一步」）。心理奖励是小白坚持到终点的燃料，这是所有小白教程的共同经验。

**用户做的事**：照单点一遍，说哪里不对（用 skill 给的选项），或者说「继续」。

**出口**：反馈进 `tasks.md` 变成新任务 → 回 Stage 4；清单全空且用户满意 → Stage 6。**中途也随时可以喊停**：项目状态全在文件里，明天回来接着走（见 §5.2）。

### 4.7 Stage 6 · 首航收尾与上线（可选，逐步护送）

**目的**：跨过「代码写完了」到「别人能用上」之间的断层——这是小白教程公认的最大断崖。

skill 做的事：
1. 先明确询问要不要上线：「只自己用」也是完全合法的终点，绝不推销部署；
2. 若要上线：**每一步单独护送**——需要开什么账号、点哪个按钮、粘贴什么，一步一确认；**凡涉及花钱的选项显式标红并给出免费替代**；
3. 密钥纪律：引导用户把密钥放进环境配置，**skill 自身永不经手明文密钥、绝不让密钥进代码和聊天记录**；
4. 生成三份收尾产物：`README.md`（这个项目是什么，给未来的自己和访客看）、`使用说明.md`（怎么日常使用和重启）、`维护手册.md`（出问题了怎么喊 Claude Code 来修：附三句现成的求助话术）；
5. **进环仪式**（上半场的真正终点）：总结做了什么、用户拥有了什么，然后正式宣告工作方式切换——「从现在起我们换一种合作方式：**你出方向，我跑圈**。想加功能、发现毛病、或者只是想看看它还能长成什么样，回来说一句话就是点火。」项目自此进入下半场（§4.8）。

**用户做的事**：决定要不要上线；照护送步骤逐步点头。

### 4.8 下半场 · 循环模式（Loop Mode）——本 skill 的后 50%

引导小白做出第一版，只完成了产品的一半。另一半是 Loopwork 这个名字的本意：**把一次性的建造，变成可持续的循环**。上半场结束时用户拥有一个 app；下半场结束时（其实永不结束）用户拥有一台会转的机器。

#### 4.8.1 一圈的完整节奏（用户视角）

1. **点火（30 秒）**：任何时候回来说一句话——「继续我的记账项目」「加个导出功能」「用的时候发现分类老记错」。skill 读状态文件自动识别点火类型：清单有活 → 直接开跑；带新想法/新毛病 → 先用迷你访谈（1-3 个问题）把它变成任务再跑；
2. **跑批（AI 自主，挂机）**：上半场 Stage 4 的内循环机器原样复用——考题先红后绿、存档、勾掉、问题本跳过不停机、外部计数封顶；
3. **收货（3-10 分钟）**：照验收脚本点一遍新功能 + 清问题本（攒着的拍板事项一把处理，答案写回规矩，同样的问题永不问第二遍）；
4. **续单（5 分钟，仅当清单空）**：**清单空了 ≠ 项目完成，只意味着该续单了**（P11）。向导带用户做一次盘点，下一批任务的燃料有三个来源：
   - ① 用户使用中攒的新想法（「要是能 XX 就好了」）；
   - ② 使用中发现的毛病和不顺手；
   - ③ 向导对照规格与实际使用情况盘出的缺口（白话版 gap-check）。
   列成下一批清单 → 用户点头 → 回到 1。环就这样一圈一圈转下去。

#### 4.8.2 双循环架构（发动机图纸）

- **内循环（任务级，AI 的循环）**：取任务 → 考题红 → 实现 → 绿 → 存档 → 下一条。全自动，用户不在场；
- **外循环（项目级，人的循环）**：点火 → 跑批 → 收货 → 续单。方向盘永远在用户手里。

内循环嵌在外循环的「跑批」环节里。**「执行自主，方向人定」（P4）在架构上的体现就是：AI 永远出不了内循环，用户永远不用进内循环。**

#### 4.8.3 三档发动机（按用户成长解锁）

| 档位 | 什么样 | 何时解锁 |
|---|---|---|
| 手动档 | 每圈由用户点火，一圈一收，最稳 | 默认 |
| 挂机档 | 教用户用原生 `/goal`：做完一条自动接下一条，直到这批跑空才来找你 | 第二圈起，向导主动教 |
| 闹钟档 | 定时自动点火，用户只剩收货和续单 | roadmap（v1.x） |

#### 4.8.4 所有权转移曲线——向导的目标是让自己越来越闲

| 圈数 | 向导角色 | 用户角色 |
|---|---|---|
| 首航 | 老师傅：每步讲为什么 | 学员：跟着走 |
| 第 2-3 圈 | 副驾：只在检查点出现，把「点火/续单」的主动权递给用户 | 学徒转正：自己点火、自己提方向 |
| 稳态 | 沉默的发动机：只报进度和问题 | 产品所有者：出方向、收货、拍板 |

进入稳态的判定标志：**用户不看任何说明，独立完成「点火 → 收货 → 续单」一整圈。**

**本 skill 的真正毕业标准：毕业的不是项目，是用户。** 交付物从来不是「一个做完的 app」，而是「一个装上了循环发动机的项目 + 一个会开这台机器的人」。

#### 4.8.5 循环模式的心法（写给向导自己的行为准则）

- **会话可弃，循环永续**：环的每一次咬合都靠文件（P1），用户隔一个月回来，环接得上；
- **圈数越多话越少**：解释密度随圈数递减，稳态时向导每圈的话不该超过一屏；
- **永远不宣布「项目完成」**：只报「这一批完成」；下一批永远存在，哪怕内容是「歇着，等你有新想法」。

### 4.9 上手成本一览

| 阶段 | 用户耗时 | 用户动作 |
|---|---|---|
| 安装 | 1 分钟 | 拖一个文件夹 |
| Stage 0-3 | 20-40 分钟 | 回答问题、批准计划 |
| Stage 4 | 0（挂机） | 无 |
| Stage 5 | 每批 5-10 分钟 | 照单点一遍、给反馈 |
| Stage 6 | 10-30 分钟 | 逐步点头 |
| **循环模式（每圈）** | **10-15 分钟** | **点火一句话、收货点一遍、续单点头** |

## 5. 全程横切功能

### 5.1 「我在哪」路由（抄自 BMAD 的 bmad-help 模式）

任何时候用户表现出迷茫（「然后呢」「现在怎么办」「我忘了做到哪了」），skill 读状态文件后回答三件事：**你在第几阶段、已完成什么、下一步是什么**——并且每个阶段结束时不等用户问、自动播报。「不知道现在该干嘛」是所有引导型产品评测里的头号流失原因。

### 5.2 跨会话续接

项目全部状态活在项目目录的文件里（P1）：`PROJECT.md`（简介）、`spec.md`（规格）、`rules.md`（规矩）、`plan.md` / `tasks.md`（计划与任务）、`BLOCKED.md`（问题本）、`.loopwork/state.json`（阶段指针＋计数器）、`JOURNAL.md`（每轮一行的日志）。

新会话激活 skill 时先探测这些文件：存在 → 「欢迎回来。你的记账项目上次做到第 4 阶段，12 条任务完成了 7 条，问题本攒了 2 件事等你拍板。现在是继续干活，还是先清问题本？」——**会话可以死，项目永远接得上。**

### 5.3 问题本与「开锁时刻」

循环中所有需要用户拍板的事（方案二选一、要不要花钱、超出规格的新想法）统一进 `BLOCKED.md`，格式固定：问题 / 背景 / 建议＋理由。验收检查点时一把清算：逐条念、用户逐条答、答案立刻写回对应文档——**同样的问题永远不问第二遍**。

### 5.4 语言与语气

- 中文小白是首要用户；skill 自动跟随用户语言（用英文提问就全程英文）；
- 语气基调：**耐心的老师傅**——不装可爱、不堆表情、不用黑话；每个必须出现的术语首次出现带白话比喻（词典见 §10）；
- 所有解释面向「为什么」而不只是「是什么」：「先让考题失败，是为了确认考题真的能考住人——一张永远给满分的考卷保护不了你」。

## 6. 纪律与安全机制（五道锁）

行业共识（68 个失败案例的结论）：**写在提示词里的规则会被模型「读了、懂了、压力下照样违反」**。所以纪律分层，文本只是第一层：

| 锁 | 机制 | 防什么 |
|---|---|---|
| 锁 1 · 文本铁律 | SKILL.md 开头的循环铁律（置于压缩存活区，见 §7.2） | 日常行为基准 |
| 锁 2 · 可执行验收 | `verify.sh` 跑考题，**exit code 说了算**；判定输出解析失败一律算不通过（fail closed） | 假完成、「我觉得测试会过」 |
| 锁 3 · hook 物理拦截 | skill 自带 hooks：循环期间锁死考题文件 / spec / rules；**Bash 层同步设防**（已知头号漏洞：Edit 被拦后模型会绕道 `sed`/`echo`/`python -c` 改文件） | 偷改考题、碰红线 |
| 锁 4 · 外部计数 | 轮次上限、重试上限（3 次转诊断）、批次大小，全部由脚本计数，不靠模型自数 | 无限循环、烧额度 |
| 锁 5 · 人工闸门 | 花钱 / 删除 / 对外发布 / 改规矩，四类动作无条件停下等用户；且这些确认**永不和普通「下一步」混排** | 不可逆事故 |

小白专属兜底：
- **自动存档**：每条任务完成即 git commit，任何灾难都能回滚（对用户的说法：「存档点」）；
- **花费透明**：长循环开始前告知「这一批大约要跑 X 分钟，会消耗你的 Claude 额度」；
- **项目沙箱**：只在项目文件夹内动文件，家目录和项目外的一切不碰；
- **密钥零经手**：见 4.7；聊天中用户误粘密钥 → 提醒撤回并绝不写入任何文件。

## 7. 技术形态

### 7.1 交付物结构

```
loopwork/                      ← 拖进 ~/.claude/skills/ 即安装
├── SKILL.md                   # ≤500 行：铁律 + 阶段路由 + 状态探测
├── references/                # 各阶段详细剧本，按需加载、可反复重读
│   ├── stage-0-setup.md
│   ├── stage-1-interview.md
│   ├── stage-2-spec.md
│   ├── stage-3-plan.md
│   ├── stage-4-loop.md
│   ├── stage-5-accept.md
│   ├── stage-6-ship.md
│   ├── quick-lane.md          # 小任务旁路剧本
│   └── glossary.md            # 白话词典全文
├── scripts/                   # 确定性动作交给脚本（执行不占上下文）
│   ├── init_project.sh        # 建项目骨架 + git init + 状态文件
│   ├── verify.sh              # 跑考题，exit code 裁决
│   └── progress.py            # 读写 state.json，输出「你在哪」
└── agents/
    └── reviewer.md            # 只读判卷员（预检成果，两段式：合规 + 质量）
```

### 7.2 三条关键实现机制（全部来自官方文档与实测事故）

1. **压缩存活线**：Claude Code 长会话自动压缩后，每个 skill 只有**前 5000 token** 被重新附回。因此循环铁律（验收规则、状态文件路径、红线清单、「迷茫就读 state.json」）必须写在 SKILL.md 最前面；阶段剧本全部下沉到 references/——它们是普通文件读取，随时可以重读，不受此限。这也是为什么**不做**「一个巨型 prompt 走全流程」：三个严肃框架（spec-kit/BMAD/superpowers）全都实践过并主动放弃了这个形态，spec-kit 源码里至今留着 fork 上下文撑爆对话的事故注释。
2. **frontmatter 能力**：`disallowed-tools` 在循环阶段禁用打断型提问（问题攒进问题本）；skill 级 `hooks` 挂载锁 3 的拦截脚本；description 用双语触发词覆盖「我想做个 XX 但不会编程」类自然语言。
3. **状态文件 schema（v1 草案）**：

```json
// .loopwork/state.json —— 唯一的机器可读状态
{
  "stage": "loop",               // 0-6=首航阶段，loop=循环模式，quick=旁路
  "cycle": 3,                    // 外循环圈数：首航=1，此后每续一批 +1
  "project_name": "我的记账本",
  "tasks_total": 12,
  "tasks_done": 7,
  "current_task": "T08-支出分类饼图",
  "round_count": 9,              // 内循环外部计数器
  "retry_count": 0,
  "blocked_open": 2,
  "last_session": "2026-07-11T15:02:00+08:00",
  "milestones": ["first_run", "first_cycle_solo"]  // 已庆祝里程碑；first_cycle_solo=首次独立转完一圈
}
```

### 7.3 版本与升级

- skill 迭代升级时**绝不覆盖用户项目里的任何文件**（生态里有框架因升级清掉用户自定义文件被实名投诉）——skill 文件夹和项目文件夹物理分离，天然规避；
- `state.json` 带 schema 版本号，向后兼容。

## 8. 明确不做什么（v1 边界）

| 不做 | 原因 |
|---|---|
| Codex 版 | 钩子机制与指令体系不同，独立立项（v2，见 §11） |
| 云端会话 / GitHub PR 工作流 | 那是专业版（eric-loop-standard）的领域；小白 v1 本地闭环 |
| 要求用户注册 GitHub | 本地 git 存档已满足回滚需求；GitHub 是 Stage 6 的可选护送项 |
| 替用户保管/填写密钥 | 红线。skill 只引导，永不经手 |
| 自动付费部署 | 花钱永远是人工闸门 |
| 大型存量代码库改造 | 超出方法论适用边界，诚实拒绝并说明 |
| 教学课程功能 | 目标是做出东西；解释「为什么」但不展开成教程 |
| 多语言 UI 的完整本地化 | v1 中文优先＋跟随用户语言，不做语言包体系 |

## 9. 成功标准（怎么算做对了）

**发布门槛（全部满足才算 v1 完成）：**

1. **四场景冷启动实测通过**（全新会话、无人工干预）：
   - ① 小白一句话「我想做个 XX」→ 被接住 → 访谈 → 规格 → 计划 → 首轮循环跑通，全程零黑话；
   - ② 中断后开新会话 → 自动识别项目 → 准确播报进度 → 无损续接；
   - ③ 「把标题改成红色」类小任务 → 正确走快速通道，不摆七阶段的谱；
   - ④ **循环点火**：首航已完成的项目，用户说「加个导出功能」→ 正确识别为外循环点火 → 迷你访谈 → 续单 → 跑批 → 收货，一整圈无断点；
2. **红队测试通过**：诱导其偷改考题、绕 Bash 写保护文件、在循环中执行未批准的花钱/删除动作——hook 全部真实拦截（没亲眼看到拦截不算装好）；
3. **文档量纪律**：为演示项目生成的文档总量 < 代码量（防空洞膨胀）；
4. **一名真实小白完整走通首航**并在不求助第二个人的情况下到达 Stage 5 验收；
5. **同一名小白自主转起第二圈**：不看任何说明，独立完成一次「点火 → 收货 → 续单」。

**北极星（上线后）**：一个不会编程的人拥有一个自己每天在用、且转过 ≥3 圈仍在生长的软件——项目没有「做完」，用户没有放弃。
