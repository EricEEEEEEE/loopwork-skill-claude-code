# Stage 5 · 验收（用户的主场）

原则：**小白不读代码，小白点鼠标。** 你负责把「验证正确性」翻译成「照着点一遍」。

## 流程

1. **预检（用户看到东西之前）**：
   - `verify.sh` 全绿确认；
   - 派只读判卷员：**读取 skill 目录下 `agents/reviewer.md` 的正文，把它作为任务指令，派一个通用型专员（只给读取类权限，不给写权限）执行**，审这一批：合规（对得上 spec？碰过红线？）+ 质量（明显隐患？）两段式；
   - 判卷结论浓缩成一句话附在验收邀请里。写代码的和判卷的不是同一个脑子。
2. **跑起来**：本地启动应用，给出能直接点开的地址/文件；启动命令记入「使用说明」备用；
3. **发验收单**：从本批任务对应的 spec 验收句生成，编号照抄：
   > 请照着点一遍（约 3 分钟）：
   > ① 打开 http://localhost:3000 —— 应看到本月总额
   > ② 点「记一笔」填 50 午饭 —— 列表立刻出现这条
   > ③ 刷新页面 —— 记录还在
   > 哪条不对告诉我编号。
4. **收反馈（强制具体化）**：用户说「不好看/不对」→ 选择题追问：「哪里？① 颜色 ② 字太小 ③ 排版乱 ④ 功能不对 ⑤ 其他」。模糊反馈是无效修复循环的根源，宁可多问一个选择题；
5. **反馈落地**：每条反馈 → tasks.md 新任务（走同样的考题流程），别当场即兴乱改；
6. **清问题本**：BLOCKED.md 有存货就逐条念：问题一句话 + 建议 + 理由 → 用户拍板 → 答案写回对应文档（规格变更进 spec，规矩变更进 rules）→ 同样的问题永不再问；
7. **庆祝里程碑**（查 state.milestones 防重复）：
   - `first_run` 首次跑通：「🎉 它活了——很多人到不了这一步」；
   - `half` 过半；`voyage_done` 首航完成；`first_cycle_solo` 首次独立转完一圈。

## 出口

- 有新任务/反馈 → 回 Stage 4；
- tasks.md 全空且用户满意 → Stage 6（首航期）或续单流程（循环模式）；
- 用户想歇 → 状态落盘，告诉他回来说「继续」就行。
