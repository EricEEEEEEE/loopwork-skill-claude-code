# Stage 3 · 摊开计划

目的：动手前把「接下来会发生什么」一次性摊开，拿到用户明确批准。**这是代码动工前的最后一道人工闸门。**

## plan.md：技术选型的白话理由

- 选型原则：最常见、生态最大、出问题最好修——不追新不炫技；参考 state.json 的 `env` 字段（Stage 0 探测的运行时）；
- 每个选择配一句白话理由：「界面用网页（HTML）因为不用装任何东西、手机电脑都能开」；
- 技术细节（目录结构、依赖清单）收进文末「技术附录」折叠区，主体永远是白话。

## tasks.md：任务清单格式

```markdown
# 任务清单

- [ ] T01 · 项目能跑起来（打开是空白页但不报错）〔考题: F0〕〔约20分钟〕
- [ ] T02 · 能记一笔支出〔考题: F1.1-F1.2〕〔约30分钟〕★首个里程碑：你能亲手记一笔
- [ ] T03 · 支出列表能显示〔考题: F1.3〕〔约20分钟〕
…
```

规则：
- 每条 = 一个**用户可感知的进步**（「做完这条你就能看到 XX」），不按技术层次拆（禁止「搭建数据层」这种条目）；
- 每条标注对应 spec 验收句编号 + 时间预估；
- ★ 标记里程碑（首个能点开的版本、核心功能闭环、全部完成）；
- 排序原则：最快让用户见到东西的路径优先（界面先行）；
- 第一批控制在 5-8 条。

## 摊牌话术

「计划好了：一共 N 步，预计第 M 步（约 X 分钟后）你就能亲眼点开第一个版本。批准我就开工；想先看到别的（比如先做界面）现在说，我重排。」

用户要求调整 → 重排 → 再次确认。**没有明确的「批准/开工/可以」不许进 Stage 4。**

## 退出条件

- [ ] plan.md + tasks.md 定稿
- [ ] 用户明确批准（记录原话进 JOURNAL）
- [ ] git 存档；state.stage → 4

→ 转场：「收到，开工。接下来一段时间不需要你，我每完成一条会报一行进度。你去喝咖啡，机器响了再回来。」
