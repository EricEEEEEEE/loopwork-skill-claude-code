# Stage 1 · 想法访谈

目的：把一句话想法变成一份双方点头的项目简介（PROJECT.md，≤300 字）。

## 提问纪律（硬规则）

- **一次只问一个问题**；用 AskUserQuestion 出选择题（2-4 个选项 + 自动的「其他」），推荐项放第一并标「（推荐）」；
- 每个问题都留「不知道，你来定」出口——用户选了就给行业常规默认值并用一句话解释为什么；
- 总量控制：一轮最多 5-8 个问题，问完立刻汇总，绝不车轮战；
- 用户答案实时写进草稿，别攒到最后。

## 六件事问题库（按需取用，别机械全问）

1. **给谁用**：就你自己（推荐，最简单）/ 你和家人朋友 / 给很多人用；
2. **解决什么问题**：现在没有它，你是怎么凑合的？最烦的一点是什么？
3. **必须有什么**：第一版哪 1-3 个功能是「没有就没意义」的？
4. **明确不要什么**：哪些常见功能你确定不要（登录？多人？手机版？）——砍掉的每一刀都让第一版来得更快；
5. **什么算成功**：一句话描述「用上它的第一个爽点时刻」；
6. **红线**：有什么数据/文件/习惯是绝对不能被打扰的？

判断哪些能推断就不问（给谁用=自己 → 跳过登录相关），把省下的问题额度用在模糊处。

## 待澄清标记（防静默假设）

草稿里任何拿不准的地方写 `【待澄清: …】`。**标记不清零，不得进入 Stage 2。** 每个标记要么追问一个问题解决，要么明确说「这条我按 XX 默认处理，不同意随时改」。

## 产出：PROJECT.md 模板

```markdown
# {项目名}

**给谁用**：…
**解决什么问题**：…
**第一版必须有**：1… 2… 3…
**明确不做**：…
**什么算成功**：…
**红线**：…
```

逐段念给用户确认（不是甩整篇），有异议当场改。

## 退出条件

- [ ] PROJECT.md 定稿且零待澄清
- [ ] 用户明确说了「对/可以/就这样」
- [ ] git 存档；state.stage → 2

→ 转场：「想法钉死了。下一步我把它翻译成『做成什么样才算对』的清单——每一条都是你将来能亲手验收的。」
