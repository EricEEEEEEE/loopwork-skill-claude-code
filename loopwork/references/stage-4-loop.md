# Stage 4 · 循环执行（内循环引擎的完整操作手册）

SKILL.md ④ 是节奏总纲（压缩后仍在），本文是细则。两者冲突时以 SKILL.md 为准。

## 一轮的完整动作（一轮 = 一条任务）

1. **取任务**：tasks.md 第一条未勾条目；`python3 .loopwork/hooks/progress.py set phase test-writing`
2. **写考题**：把该任务标注的 spec 验收句直译成自动测试。考题原则：
   - 测「用户可见的结果」，不测实现细节；
   - 用假数据替身，不连真实外部服务、不联网；
   - 考题文件统一放 `tests/`；
3. **亲眼跑红**：运行考题，确认失败，失败摘要记入 JOURNAL（「红」是证据，证明考卷真的能考住人）。**先把失败考题单独存档**（commit 里只有测试文件）；
4. **锁围栏**：`progress.py set phase implementing` —— 此刻起 guard_edits 物理锁死 tests/、spec.md、rules.md；
5. **写实现**：最小改动让考题变绿；不写投机功能、不顺手重构无关代码；
6. **验证**：`bash .loopwork/hooks/verify.sh`
   - exit 0 = 绿；非 0 = 继续修；
   - 长输出必须 `> .loopwork/logs/round-N.log 2>&1` 然后 `tail -20`，绝不把整版日志倒进对话；
7. **收轮**：绿 → `git commit -m "存档: T{编号} {任务名}"` → tasks.md 勾掉 → `progress.py bump-round` → JOURNAL 追加一行：`T02 ✅ 2 红→绿 | 备注`
8. **进度播报**（一行）：「✅ 3/12 · 记一笔完成 · 下一条：支出列表」

## 问题本（BLOCKED.md）

触发：方案二选一、要花钱、要装重量级依赖、触碰 rules.md、发现规格矛盾、超出规格的新想法。

```markdown
## B03 · 图表库选择
- 问题：饼图需要图表库
- 背景：手写约 200 行；引库 3 行但多一个依赖
- 建议：引 Chart.js（最常见），理由：省 2 小时且更稳
- 状态：待拍板
```

写完**立刻跳过取下一条**，绝不停机干等。被跳过的任务在 tasks.md 标 `〔卡·B03〕`。

## 停批条件（满足任一即停，进 Stage 5）

- 做满一批（state.batch_size，默认 5 条）；
- 撞到 ★ 里程碑；
- 连续 2 条任务受阻；
- round_count 达上限（默认 20，外部计数器管着，不靠你自己数）。

停批汇报格式：本批 N 对 M 点名 + 考题总数与结果 + 问题本新增 + 下一批预告。

## 失败分级（同一任务）

1 次失败 → 正常修；2 次 → 换思路修；**3 次 → 停止重试，转诊断模式**：读完整日志 → 定位根因（环境？依赖？考题写错？）→ 修根因 → 只再试一次 → 仍败 → 写 BLOCKED.md（含已试方案清单）跳过。

## 挂机批模式（可选，用户同意后开启）

- 开：`touch .loopwork/batch.flag` —— Stop 钩子会在这批未跑完时自动把你顶回去继续干；
- 关：批完自动删 flag；用户随时说「停」→ `rm .loopwork/batch.flag`；
- 开启前必须告知：「这一批大约 X 分钟，期间会持续消耗你的 Claude 额度，中途喊停随时生效。」
