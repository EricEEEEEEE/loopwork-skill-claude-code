# 下半场 · 循环模式（本 skill 的后 50%，永不散场）

上半场结束时用户拥有一个 app；下半场用户拥有一台会转的机器。**外循环：点火 → 跑批 → 收货 → 续单。项目不是做完的，是转起来的。**

## 点火类型识别（用户回来说的第一句话）

| 用户说 | 类型 | 动作 |
|---|---|---|
| 「继续」「接着干」 | 常规点火 | 清单有活 → 直接跑批；清单空 → 续单流程 |
| 「加个 XX 功能」 | 带燃料点火 | 迷你访谈（1-3 问）→ 写成任务 + 考题句 → 确认 → 跑批 |
| 「XX 坏了/不对」 | 报修点火 | 复现 → 写成修复任务（先写考题钉住 bug）→ 跑批 |
| 「我在哪」「然后呢」 | 路由 | progress.py card 播报 |
| 「不想要 XX 了」 | 减法点火 | 确认影响面 → 移除任务 + 更新 spec |

带燃料点火的迷你访谈上限 3 个问题：这功能给谁用/什么样算做对了/多急（按现场换成更有信息量的维度也行）。答案直接写成 spec 增量（新增 F 编号）+ tasks 条目，**不重写整份规格**。

登记的三条硬规则：
1. **动笔前查相位**：`progress.py get phase` 非 `test-writing` 先复位（上一圈可能中断在实现期，不复位会被围栏拦）；
2. **登记完即存档**：spec 增量 + 新任务写完，立刻 `git commit -m "存档: 登记 F{n} {功能名}"`——登记也是记忆，不能只活在工作区；
3. **确认是硬闸门**：登记念给用户、用户明确点头后才许跑批——分量等同 Stage 3 的计划批准，没有点头一行实现代码都不许写。

## 一圈的节奏（外循环）

1. **点火**（用户，30 秒）：识别类型 → 必要时迷你访谈 → 确认这批要干什么；
2. **跑批**（你，挂机）：内循环引擎原样运转（SKILL.md ④ / stage-4 剧本）；
3. **收货**（用户，3-10 分钟）：stage-5 验收流程 + 清问题本；
4. **续单**（仅当清单空，5 分钟）：见下。
5. `progress.py bump-cycle`，圈数 +1。

## 续单流程（清单空了 ≠ 项目完成，= 该续单了）

盘点三个燃料来源，各出 1-3 条候选：
1. **用户攒的新想法**：「用了这段时间，有没有『要是能 XX 就好了』的时刻？」
2. **使用中的毛病**：「哪里用着别扭、慢、丑、绕？」
3. **向导盘缺口**（白话版 gap-check）：对照 spec 与实际使用——规格里写了没做全的？边界情况（数据多了、输错了、换设备）？该有说明没说明的？

汇成候选清单（每条一句话 + 值不值得的理由）→ 用户挑 → 写入 tasks.md → 回到点火。
用户说「暂时不想加了」→ 完全合法：「机器停在这，随时回来说一句话就转。」状态落盘，圈不散。

## 三档发动机（按成长解锁）

| 档位 | 什么样 | 何时提 |
|---|---|---|
| 手动档 | 每圈用户点火，一圈一收 | 默认 |
| 挂机档 | 批模式（batch.flag + Stop 钩子）或教用户原生 `/goal`：「输入 /goal 这批任务全部完成且考题全绿——它会自动一条接一条干到批空」 | 第 2 圈起主动教一次 |
| 闹钟档 | 定时自动点火 | v1 不做，用户问起就说在路线图上 |

## 所有权转移（你的目标是越来越闲）

- 第 2-3 圈：把「点火/续单」的主动权递出去——「下次你可以直接说『继续』，不用等我问」；解释密度减半，只在检查点出现；
- 稳态（用户能不看说明独立转完一圈）：庆祝 `first_cycle_solo`（只此一次）；此后你每圈的话不超过一屏：进度卡 + 验收单 + 问题本，没了；
- **永不宣布项目完成**：只报「这一批完成」。下一批永远存在，哪怕内容是「歇着，等你有新想法」。
