# Stage 2 · 白话规格 + 项目规矩

目的：把简介展开成「做成什么样才算对」的验收清单（spec.md），并立好红线（rules.md）。

## spec.md：验收句格式（本 skill 的命根子）

每条功能写成**用户视角、可亲手验证**的句子。禁止技术黑话。

✅ 好：「打开页面能看到本月支出总额；点『记一笔』，填金额和分类，保存后列表里立刻出现这条；刷新页面它还在。」
❌ 坏：「实现 CRUD 接口和数据持久化层。」

这些句子的三重身份：① 用户现在读它点头；② Stage 4 把它直译成考题；③ Stage 5 用户照它点一遍验收。**一句顶三处用，所以每句都要能落到具体动作和具体可见结果。**

```markdown
# {项目名} · 规格

## F1 {功能名}
验收：1. …（动作 → 可见结果） 2. …
## F2 …

## 边界（明确不做）
- …
```

## 篇幅纪律（防空洞膨胀）

规格 ≤ 100 行、每个功能 ≤5 句验收。写得越长不代表越专业——小白没能力鉴别注水文档，膨胀等于把验收环节变成走过场。

## 确认节奏

逐节展示（每节 ≤300 字）→ 「这节对吗？」→ 下一节。任何肯定回应即推进，不逼用户精确措辞。

## rules.md：项目规矩（默认六条 + 用户红线）

```markdown
# 项目规矩（碰这些之前必须先问我）
1. 不删除我任何既有文件
2. 不花任何钱
3. 不对外发布任何东西
4. 密钥和密码永远不写进代码
5. 考题写好后，实现期间不许改考题
6. 每完成一条任务立刻存档
7+. {访谈问出的用户红线}
```

向用户解释：这不是客套，是围栏真的在执行的清单——第 5 条由 guard_edits 物理拦截，第 1/3 类动作由 guard_bash 拦截。

## 退出条件

- [ ] spec.md 逐节确认完毕、≤100 行
- [ ] rules.md 用户过目
- [ ] git 存档；state.stage → 3

→ 转场：「标准定了。下一步我出施工计划和任务清单，**你批准之前我一行代码都不会写**。」
