# 快速通道（小任务旁路）

判定（三条全满足才走这里）：≤3 个文件、预计 ≤30 分钟、不改变项目结构。例：改文案、调颜色、改一个数值、修一个明确的小 bug。

**为什么存在**：拿七阶段去处理「把标题改成红色」是大锤砸坚果，会把用户吓跑。

## 迷你流程（一个会话内走完）

1. **确认**（1 问）：复述理解——「你要的是把 X 改成 Y，对吗？」
2. **迷你计划**（≤3 行）：改哪个文件、怎么验证。有 `.loopwork/` 的项目照常受围栏保护；
3. **动手**：改 → 有考题体系就跑 `verify.sh`，没有就给出「照着点一遍」的单条验收；
4. **收尾**：git 存档（项目有 git 时）+ 一行汇报。

## 升级出口

动手后发现比想的大（要动 >3 文件 / 要改结构 / 牵连别的功能）→ **立刻停**，说：「这活比看起来大，硬改会埋雷。建议转正式流程（约多花 X 分钟），或者先只做安全的一小半。你选。」绝不静默扩大战线。

## 无项目场景

用户拿着零散文件求小改（没有 .loopwork/）→ 照常服务，但结尾提一句：「如果这个东西你会持续改，我可以给它建个正式项目（存档+围栏+清单），以后每次改都更稳。要吗？」不强推。
